## 工作流：创建新技能

### 第1步：规划与设计（参考文档第2章）

**1.1 定义具体用例**
在编写任何代码前，确定2-3个具体用例。

**好用例示例：**

```markdown
用例：项目冲刺规划
触发条件：用户说"帮我规划这个冲刺"或"创建冲刺任务"
步骤：
1. 通过Linear MCP获取当前项目状态
2. 分析团队速度和容量
3. 建议任务优先级
4. 在Linear中创建带有正确标签和估算的任务
结果：创建了完整规划的冲刺任务
自问：
• 用户想要完成什么？
• 这需要哪些多步骤工作流？
• 需要哪些工具（内置或MCP）？
• 应该嵌入哪些领域知识或最佳实践？
```

**1.2 选择技能类别**
根据文档，常见技能类别有：

**类别1：文档和资产创建**
- 用于：创建一致的、高质量的产出（文档、演示文稿、应用、设计、代码等）
- 关键技术：嵌入样式指南、模板结构、质量检查清单
- 示例：frontend-design技能

**类别2：工作流自动化**
- 用于：受益于一致方法的多步骤流程
- 关键技术：分步工作流、验证关卡、模板、迭代改进循环
- 示例：skill-creator技能

**类别3：MCP增强**
- 用于：增强MCP服务器提供的工具访问的工作流指导
- 关键技术：协调多个MCP调用、嵌入领域专业知识、提供上下文
- 示例：sentry-code-review技能（来自Sentry）

**1.3 定义成功标准**

**定量指标：**
- 技能在90%的相关查询中触发
- 在X次工具调用中完成工作流
- 每个工作流0次失败的API调用

**定性指标：**
- 用户不需要提示Claude下一步
- 工作流无需用户修正即可完成
- 跨会话结果一致

### 第2步：技能结构（参考文档第2章）

**2.1 文件夹结构**

```
your-skill-name/           # 必须使用kebab-case（小写字母加连字符）
├── SKILL.md              # 必须精确使用此文件名（区分大小写）
├── skill.yaml            # 必填：机器可读元数据（用于工具链）
├── scripts/              # 可选：可执行代码（Python、Bash等）
│   └── validate.py
├── references/           # 可选：按需加载的文档
│   └── api-patterns.md
└── assets/               # 可选：输出中使用的模板、字体、图标
```

**关键规则：**
- 文件夹名：必须使用kebab-case（如`notion-project-setup` ✅）
  - 不允许：空格（`Notion Project Setup` ✗）
  - 不允许：下划线（`notion_project_setup` ✗）
  - 不允许：大写字母（`NotionProjectSetup` ✗）
- 不要在技能文件夹中包含`README.md`
  - 所有文档放在`SKILL.md`或`references/`中
  - 注意：通过GitHub分发时，您仍然需要仓库级的README给人看
- `skill.yaml` 必须与 SKILL.md 同级，用于工具链自动读取元数据
  - 详见 `references/output-spec.md` 中的 skill.yaml 规范

**2.2 YAML Frontmatter（最重要的部分）**

Frontmatter是系统决定是否加载您技能的方式。

**最小必需格式：**

```yaml
---
name: your-skill-name      # 必须：kebab-case，与文件夹名匹配
description: 技能功能描述和何时使用它 # 必须：包含功能和触发条件
---
```

**字段要求：**
- `name`（必需）：
  - 仅限kebab-case
  - 无空格或大写字母
  - 应与文件夹名匹配
- `description`（必需）：
  - 必须包含两者：
    1. 技能做什么
    2. 何时使用它（触发条件）
  - 少于1024个字符
  - 无XML标签（`<` 或 `>`）
  - 包含用户可能说的具体任务
  - 如果相关，提及文件类型
- `license`（可选）：
  - 如果使技能开源则使用
  - 常见：Apache-2.0
- `compatibility`（可选）：
  - 1-500个字符
  - 指示环境要求：例如目标产品、需要的系统包、网络访问需求等
- `metadata`（可选）：
  - 任何自定义键值对
  - 建议：author、version、mcp-server
  - 示例：

```yaml
metadata:
  author: 您的公司名
  version: 1.0.0
  mcp-server: your-server-name
```

**安全限制（Frontmatter中禁止）：**
- XML尖括号（`< >`）
- 名称中包含"claude"或"anthropic"的技能（保留字）
- 原因：Frontmatter出现在系统的提示中。恶意内容可能注入指令。

### 第3步：编写技能内容

**3.1 编写描述字段**

根据渐进式披露原则，描述字段提供足够信息让Claude知道何时使用每个技能，而无需将所有内容加载到上下文中。

**结构：** `[功能] + [何时使用] + [关键能力]`

**好描述示例：**

```yaml
# 好 - 具体且可操作
description: 分析Figma设计文件并生成开发人员交接文档。当用户上传.fig文件、询问"设计规范"、"组件文档"或"设计到代码交接"时使用。

# 好 - 包含触发短语
description: 管理Linear项目工作流，包括冲刺规划、任务创建和状态跟踪。当用户提到"sprint"、"Linear任务"、"项目规划"或要求"创建工单"时使用。

# 好 - 清晰的价值主张
description: PayFlow的端到端客户入职工作流。处理账户创建、支付设置和订阅管理。当用户说"为新客户入职"、"设置订阅"或"创建PayFlow账户"时使用。
```

**坏描述示例：**

```yaml
# 坏 - 太泛泛
description: 帮助处理项目

# 坏 - 缺少触发条件
description: 创建项目工作区

# 坏 - 使用XML标签（不允许）
description: 处理<重要>项目数据</重要>
```
