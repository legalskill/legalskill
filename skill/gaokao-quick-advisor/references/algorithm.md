# 核心算法决策文档

> 推荐引擎的算法逻辑与设计原则完整说明。
> **Agent 在调用 recommender.py 前必须通读此节**，以正确理解输出结果。

## 概率模型：Logistic Sigmoid

```
prob = 100 / (1 + e^(K × (ratio - 1)))
其中:
  ratio = user_rank / admit_rank
  K = 2.5（斜率系数）
  admit_rank = 该校在目标省近年的最低录取位次
  user_rank  ≈ 越接近 admit_rank，ratio → 1，prob → 50%
```

**为什么是 logistic？**
- 录取行为本质上是位次竞争。ratio=1（用户位次 = 学校最低位次）时，概率=50%（五五开）。
- ratio 偏离 1 时概率非线性衰减/增长，更符合实际——位次差1位和位次差100位的意义完全不同。
- 线性插值（旧模型）会在分数差距极大时仍给出虚假概率，误导用户。

**为什么 K=2.5？**
- K 控制曲线陡峭度。K=2.5 使得 冲/稳 边界（ratio≈1.302）处概率≈32%，稳/保 边界（ratio≈0.730）处概率≈68%，确保三个档位覆盖合理的概率区间。
- K 过小（如1.0）会模糊 冲/稳/保 三档的区分度；K 过大（如5.0）会使中间档位内容过少。

## 档位划分标准

**默认阈值**（本科场景适用）：

| 档位 | 概率区间 | 含义 | ratio范围 |
|------|---------|------|-----------|
| 🟢 冲刺 | [5%, 32%) | 需要运气，但有机会 | ratio ∈ (1.302, ~3.0] |
| 🟡 稳妥 | [32%, 68%) | 合理区间，正常机会 | ratio ∈ [0.730, 1.302] |
| 🔵 保底 | [68%, 100%] | 非常安全 | ratio < 0.730 |

**动态调整阈值**（专科场景适用）：

当分数低于本科批分数线时，logistic 概率区间被数据上限压缩（最高录取位次有限），因此 `专科为主，本科为辅` 和 `纯专科方案` 使用调整后的阈值：
- 保底 ≥ **50%**（ratio ≤ 1.0，即用户位次 ≤ 学校最低位次）
- 稳妥 [32%, 50%)
- 冲刺 [5%, 32%)（不变）

此举确保排名靠后的用户也能获得合理数量的保底推荐。

> 注：初始概率为连续值，四舍五入到一位小数后判断档位。例如 31.95% 四舍五入为 32.0% → 归属「稳妥」档。

## 冲刺档概率底限：5%

**规则**：所有概率 < 5% 的学校从推荐列表中排除。

**理由**：
- 本技能的目标是提供**可操作的参考建议**，不是「我想上清华」的闲聊。
- < 5% 的学校本质上不存在录取可能性（需要尝试 20 次以上才可能一次命中）。
- 列在列表中会降低信息的信噪比，浪费用户的注意力。
- 此阈值为设计决策，非临时参数——不得随意调低。

**约等于 5% 的学校特征**：ratio ≈ 2.0（即 user_rank 约为 admit_rank 的 2 倍）。对于 560 分历史类（rank≈18,388），约等于 admit_rank ≈ 9,200 的学校。

## 采样策略：等距采样（Strided Sampling）

```python
def _strided_sample(items, n):
    """从已排序列表中取 n 个等距样本，确保覆盖全范围。"""
    if len(items) <= n:
        return items
    step = len(items) / n
    return [items[int(i * step)] for i in range(n)]
```

**为什么不用取顶部/底部？**
- 取顶部（最高概率）：全部集中在档位边界附近，看不到远端学校。
- 取底部（最低概率）：冲到概率最低的边界，全部接近 5%，失去梯度信息。
- 等距采样：从 5% 到 32%（冲档）均匀铺开，用户看到完整的概率梯度。

**排序键**：各档位内按概率升序排列（chong: 5%→32%, wen: 32%→68%, bao: 68%→100%）。然后步长采样使推荐列表呈现从「最难」到「最易」的自然梯度。

## 展示规则：×2

`target_slots = province_slots × 2`（如河北 96 位 → 展示 192 校）。

**理由**：用户实际可用志愿位是 province_slots 个（河北 96 个「专业+院校」位），但推荐时展示双倍数量可以让用户有更宽泛的选择。用户无法填满所有学校，但可以从中挑选自己感兴趣的进一步研究。

## 四场景过滤策略（本科/专科混合推荐）

> 引擎根据分数与本科线的关系自动选择推荐方案，避免本科和专科的学校混在一起导致推荐不合理。

**场景划分**（以 `diff = 用户分数 - 本科批分数线` 为判断依据）：

| 场景 | 条件 | 策略 | 冲 | 稳 | 保 |
|------|------|------|:-:|:-:|:-:|
| 纯本科方案 | diff ≥ 20 | 全部推荐本科院校 | 本科 | 本科 | 本科 |
| 本科为主，专科为辅 | 0 ≤ diff < 20 | 本科占主体，专科只进入保底 | 本科 | 本科 | 本科+专科 |
| 专科为主，本科为辅 | -30 ≤ diff < 0 | 专科占主体，本科只进入冲刺 | 本科+专科 | 专科 | 专科 |
| 纯专科方案 | diff < -30 | 全部推荐专科院校 | 专科 | 专科 | 专科 |

**阈值说明**：
- **+20 分**：超过本科线 20 分以上时，保底档有充足的本科学校可选，无需专科兜底
- **0 分（本科线）**：刚好压线时分入「本科为主」档，保底保留专科作为安全网
- **-30 分**：低于本科线 30 分以内，本科冲刺仍有理论可能（征集志愿等），但主体转向专科
- **低于 -30 分**：完全放弃本科方向，专注专科推荐

**场景标签输出**：
- 报告中在志愿模式描述下方显示 `📐 **推荐策略：{场景名称}**`
- 终端 stderr 输出中在统计行尾部显示 `[{场景名称}]`

**动态保底阈值**：
- 专科场景（`专科为主`、`纯专科`）的 `bao_threshold` 为 50% 而非默认的 68%
- 原因：排名靠后的用户，logistic 概率被数据上限压缩，68% 阈值会导致保底永远为空

## 数据安全

所有 `data/` 目录下的 JSON 文件**禁止**上传到任何外部网络服务（包括 IMA 知识库、腾讯文档、外部 API 等）。仅可按需读取分析。
